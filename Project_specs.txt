**Persistia Ledger Specification**  
**Version:** 0.7.0
**Date:** March 26, 2026
**Purpose:** A decentralized, Mina-style succinct ledger for persistent shared state (voxel worlds, asset ownership, transfers, and simple programmable rules). Runs across independent Cloudflare accounts with no single point of failure.

> **Note:** This is a testnet. PERSISTIA tokens are for testing purposes only and have no monetary value.

### 1. High-Level Overview
- **Goal**: Provide a ledger that is:
  - Real-time (sub-100 ms updates via WebSockets)
  - Succinct (entire history verifiable from one ~5–20 KB proof)
  - Decentralized (any number of independent Cloudflare accounts can participate as nodes)
  - Permanent (canonical state survives any Cloudflare account deletion/ban via off-platform anchoring)
- **Core invariants**:
  - Every state transition is cryptographically signed and verifiable.
  - New nodes or reconnecting clients sync in milliseconds via the latest succinct root.
  - No central operator — the ledger lives as long as ≥1 honest node is online.
- **Not a full L1 blockchain**: No mining, no gas fees inside the ledger. Settlement/anchoring is offloaded to Berachain (EVM L1).

### 2. Architecture
```
Clients / Game Nodes
   ↓ WebSocket (or HTTP)
Cloudflare Node (any account) ── gossip (HTTP + p2pcf WebRTC) ── Cloudflare Node
   │
   Durable Object (one per logical "shard"/zone)
      ├── In-memory + SQLite: current state + event log
      ├── Validator (signature + rule checks)
      └── Periodic prover (Merkle or SP1 recursive)
            ↓
         Succinct Root → Berachain (immutable anchor)
```

- **Node = one Cloudflare account** (deploy the same code via one-click Wrangler).
- **Shard = one Durable Object** (spatial or logical partitioning; unlimited shards).
- **Gossip network**: Any-to-any; nodes discover each other via a public list or manual `addNode()` call.

### 3. Data Model (inside each Durable Object)

- All mutations are atomic inside the DO.

### 4. APIs (exposed by every node)
**WebSocket (real-time, via `/ws` path)**
- `join` → returns latest state snapshot (all blocks, players)
- `submit` → signed action (place/break/transfer/craft/call)
- Broadcasts deltas to all connected clients + gossips to other nodes

**HTTP endpoints** (see README.md for full API reference)

*Consensus & DAG*
- `GET /dag/status` — chain status (round, active nodes, finalized seq)
- `GET /dag/vertices?round=N` — vertices at a specific round
- `POST /event` — submit a signed event
- `GET /sync?after=N` — fetch finalized events after sequence N
- `GET /state` — full state snapshot
- `POST /addNode` — register a gossip peer
- `GET /gossip/sync` — pull vertices from peer
- `GET /network` — node identity and capabilities
- `POST /join` — register an external node with seed shards

*Wallet & Tokens*
- `GET /wallet/info?pubkey=X` — account info and balances
- `GET /wallet/balance?address=X` — token balances
- `POST /wallet/faucet` — mint testnet tokens (no monetary value)

*Smart Contracts*
- `POST /contract/deploy` — deploy WASM contract
- `POST /contract/call` — call contract method
- `GET /contract/query` — read-only contract query
- `POST /contract/app/upload` — upload frontend files to a contract
- `GET /apps` — list deployed on-chain apps
- `GET /app/{address}/` — serve on-chain app frontend

*ZK Proofs & State Verification*
- `GET /proof/zk/status` — ZK proof chain status
- `POST /proof/zk/submit` — submit a ZK proof
- `GET /proof/zk/chain` — full IVC proof chain with public values
- `GET /proof/generate?key=X` — Merkle inclusion/non-inclusion proof
- `GET /proof/commitment` — current state root and entry count

*Light Client*
- `GET /headers/latest` — latest block header with BFT commit certificate
- `GET /headers?after=N` — block headers after block N

*Validators, Anchoring, Cross-Shard, MPP, SQL*
- `POST /validator/register` — register as validator (with PoW)
- `GET /anchor/latest` — latest state anchor
- `POST /notes/create` / `POST /notes/consume` — cross-shard notes + nullifiers
- `GET /mpp/info` — Machine Payment Protocol info
- `GET /query?q=SELECT...` — read-only SQL against node's SQLite
- `GET /schema` — list all tables and indexes

### 5. Consensus & Gossip
- **Bullshark DAG-BFT**: Validators produce vertices in rounds (12s default, adaptive via EIP-1559-style parameter adjustment). Leader-based commit rule with quorum-gated advancement.
- **Quorum**: 2f+1 of active validators must include a vertex in a round for it to advance. Leader commit requires seeing quorum of vertices referencing the leader's vertex.
- **Catch-up mode**: Nodes more than 20 rounds behind process 100 commits per cycle to accelerate sync.
- **Gossip protocol**:
  - Rate-limited vertex synchronization via `/gossip/sync` endpoint.
  - Returns batched vertices, commits, and ZK proof metadata with adaptive network parameters.
  - Periodic sync pulls missing vertices by round.
- **Conflict resolution**: DAG structure ensures total ordering via topological sort of committed vertices. Invalid events rejected before storage/gossip.
- **No fork risk for anchored state**: The canonical history is the latest Berachain-anchored root.

### 6. Succinctness Layer (Mina-style)
**Phase 1 (current PoC — already implemented in v0.1 code)**:  
- SHA-256 Merkle root of `state + ownership + events` every 5 minutes.  
- New nodes download root + latest snapshot → verify hash.

**Phase 2 (next upgrade — 1–2 days work)**:  
- Replace Merkle with **SP1 zkVM recursive proof**.  
- Circuit proves: “This batch of transitions is valid given previous root (ownership rules + contract execution).”  
- Proof size remains constant (~few KB) regardless of history length.  
- Verification is <10 ms on any node.

**Phase 3 (future)**: Full recursive composition (SP1 recursion + optional o1js/Mina bridge).

### 7. Smart Contracts / Programmable Rules
- **WASM runtime** (implemented): Contracts written in Rust, compiled to `wasm32-unknown-unknown`, deployed as signed events. NEAR-style register-based ABI with deterministic fuel metering (no floats, no WASI).
- **Rust SDK** (`persistia-sdk`): Provides `storage_read`, `storage_write`, `set_return`, cross-contract calls.
- **CosmWasm compatibility** (`cosmwasm-compat`): Port CosmWasm contracts by swapping imports and rebuilding.
- **Token standard** (`token-standard`): PTS-20/PTS-721 unified standard supporting both fungible tokens and NFTs.
- **Oracle contracts** (`example-oracle`): Request external API data with BFT-aggregated callbacks.
- **On-chain apps**: Full-stack frontends served directly from contract state at `/app/{address}/`.
- **Game world**: Minecraft-lite voxel game with craftable block types (house, wall, bridge) running entirely on-chain.
- **Future**: EVM compatibility via SP1-CC (prove real Solidity calls off-chain).

Contracts are executed inside the proof → fully verifiable, no trust in the node.

### 8. Anchoring & Persistence
Every 5–10 min (or on major events):
1. Compute succinct root/proof.
2. Post `{root, proof, metadata}` to Berachain as HYTE-encoded calldata.
3. Optionally write structured data to HyberDB contract for on-chain queryability.

The ledger’s canonical state = latest valid Berachain anchor. Any node can bootstrap from it.

### 9. Security & Attack Mitigation
- **DDoS**: Cloudflare automatic edge mitigation + decentralization (must hit every node simultaneously).  
- **Spam/Sybil**: Per-pubkey rate limits inside DO + optional PoW on events.  
- **Invalid state**: Signature + rule checks reject before storage/gossip.  
- **Eclipse**: Multiple gossip paths + periodic root pull from Berachain anchor.
- **Node deletion**: Irrelevant — any other node + Berachain anchor keeps the ledger alive.  
- **Key compromise**: Short-lived session keys recommended; full key rotation possible.

### 10. Deployment & Scaling
- **One-click node**: `wrangler deploy` (same code on any Cloudflare account).  
- **Cost**: $0 (free tier for <10 players) → $5–20/month per active node (WebSocket hibernation keeps it cheap).  
- **Scale**: Unlimited nodes; spatial sharding (one DO per zone) for 1000+ players.  
- **Monitoring**: Each node logs root hashes; public dashboard can aggregate them.

### 12. Roadmap: Appchain Rollup Framework

Persistia's next phase is evolution into an **appchain rollup framework** — enabling anyone to deploy a customizable BFT chain with a single command, secured by restaked tokens, with built-in cross-chain oracle and automation services.

#### Phase 1: PoS Staking Module (~200 LOC)
Convert the current reputation-based validator model to Proof-of-Stake:
- Replace PoW admission with token staking (lock tokens → become validator)
- Stake-weighted quorum (existing reputation-weighted quorum logic, swap weight source)
- Unbonding periods (7-28 day configurable timelock before stake withdrawal)
- Slashing for equivocation (extend existing `recordEquivocation()` to burn staked tokens)
- Delegation support (delegators earn proportional rewards, validators set commission %)
- Validator set rotation (committee selection for large validator sets)

Key changes:
- `ValidatorRecord`: add `stake_amount`, `bonded_at`, `unbonding_deadline`, `delegators`
- `quorumForActiveSet()`: weight by stake instead of reputation
- New `staking_events` table for audit trail
- New `/validator/stake`, `/validator/unstake`, `/validator/delegate` endpoints

#### Phase 2: Multi-Chain Anchor Configuration (~100 LOC)
Extend state anchoring beyond Berachain to any EVM-compatible chain:
- Generalize `BerachainConfig` → `ExternalChainConfig[]` (chain_id, rpc_url, contract_address)
- Support Ethereum mainnet, Arbitrum, Base, Optimism, Polygon, etc.
- Add calldata compression (brotli/zlib) for lower anchor costs
- Monitor target chain finality before marking anchors "confirmed"
- Anchor to multiple chains simultaneously for redundancy

#### Phase 3: Outbound Transaction Execution (~300 LOC)
Enable Persistia validators to submit transactions to external chains:
- Multi-chain RPC manager (Ethereum, Berachain, Solana, Cosmos IBC)
- Transaction building per target chain (EVM calldata encoding, Solana instructions)
- Shared key management for outbound signing (threshold signatures or HSM integration)
- Nonce management across chains (prevent stuck/duplicate transactions)
- BFT consensus on outbound transactions (N/M validators must agree before submission)

This is the foundation for cross-chain oracle delivery and automation.

#### Phase 4: Cross-Chain Oracle Services (~150 LOC)
Extend the existing oracle system to serve external chains (Chainlink CRE competitor):
- **Oracle data feeds**: Validators fetch data, aggregate via BFT consensus, post results to target chains
- **Payment model**: Charge per oracle request (settle via token transfer or MPP)
- **Source reputation**: Weight data sources by historical accuracy
- **Fallback sources**: Try alternative URLs/feeds if primary fails
- **Historical data**: Return price history / TWAP over configurable windows
- **SLA guarantees**: Timeout compensation, guaranteed precision, freshness bounds
- **ZK attestation**: Each oracle result accompanied by SP1 proof that N/M validators agreed

#### Phase 5: Cross-Chain Automation (~200 LOC)
Extend triggers to execute on external chains:
- Smart contracts on other chains register trigger conditions with Persistia
- Persistia monitors conditions via oracle polling (e.g., "when ETH < $2000")
- When condition met, validators reach BFT consensus and submit transaction to target chain
- Use cases: liquidations, rebalancing, yield harvesting, governance execution
- Payment: pre-funded balance on Persistia or per-execution fees

#### Phase 6: Gateway Contracts (Solidity)
Deploy Persistia gateway contracts on target chains for inbound request listening:
- `PersistiaGateway.sol`: Receives oracle requests, trigger registrations, message relay
- Emits events that Persistia validators watch via oracle system
- Verifies Persistia BFT commit certificates for trustless result delivery
- Optional: Verify SP1 ZK proofs on-chain for fully trustless bridging

#### Phase 7: Restaking Security Layer (~200 LOC + Solidity)
Allow appchains to be secured by restaked tokens from external chains:
- Deploy `PersistiaStaking.sol` on Berachain/Ethereum
- Stakers lock ETH, BERA, or any ERC-20 in the contract
- Persistia validators observe the lock via oracle (multi-node consensus on L1 state)
- Validator admitted with weight proportional to restaked amount
- Slashing: Persistia submits ZK proof of misbehavior to L1 staking contract
- Shared security: One restaked position can secure multiple Persistia appchains simultaneously
- Compatible with EigenLayer/Symbiotic AVS model

#### Phase 8: Appchain Template & SDK
Make deploying a custom Persistia appchain turnkey:
- `persistia init` CLI tool: generates wrangler.toml, consensus config, staking params
- Configurable parameters: round time, quorum threshold, staking token, anchor chain(s)
- Pre-built modules: DEX, lending, NFT marketplace, DAO governance
- One-command deploy: `persistia deploy --chain my-appchain --stake-token BERA --anchor ethereum`
